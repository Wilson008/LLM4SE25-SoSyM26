###  Transformation Strategy for Instance 1 -> Instance 2

1. **All actions must now have an explicit return type** placed *immediately after* `action`.  
   - In Instance 1:  
     - `action updateName(...)` -> becomes `action String updateName(...)`  
     - But wait - **Grammar 2** says:  
       ```xtext
       IsisAction:
         => (annotations+=XAnnotation*
         'action') type=JvmTypeReference name=ID '{'
         (features+=IsisActionFeature | parameters+=IsisActionParameter | events+=IsisEvent)*
         '}';
       ```
       -> So return type comes **right after `action`**, *before* name.  
       -> Signature parameters are **moved inside** the `{}` as `IsisActionParameter`.

2. **All parameters in actions must be rewritten** from Java-style in-parentheses to:  
   ```dsl
   parameter type=FullJvmFormalParameter '{ ... }'
   ```
   - even if no features.

3. **Helper methods like `default0UpdateName`, `validateUpdateName` are *not* standalone actions anymore** - in Apache Isis, they are *derived* features (`default`, `validate`), and in Grammar 2, they become `IsisActionFeature`s inside the action block (e.g., `default { ... }`, `validate { ... }`).  
   So instead of separate actions, we'll have:
   ```dsl
   action String updateName '{'
     parameter String newName '{ ... }'
     default { getName() }
     validate { ... }
   '}'
   ```

4. **Keep all annotations, whitespace, indentation, empty lines, and comments unchanged** - per instructions.

5. **Remove any top-level `event` definitions unless they're needed elsewhere** - but in Grammar 2, `IsisEvent` still exists and is allowed in action bodies.

6. **No `IsisEvent` in Instance 1 is used inside blocks**, so keep top-level `event UpdateNameDomainEvent`.

7. **Do not introduce `IsisCollection` or `IsisRepository` outside where they weren't** - keep `repository { ... }` as is (Grammar 2 still supports it, and it's unchanged in rule body).
