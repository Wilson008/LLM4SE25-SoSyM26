###  Important Note on `event`:
- In Grammar 1, `IsisEntity` allowed `events+=IsisEvent` *as direct child*, e.g.:
  ```dsl
  entity X { 
    event MyEvent   //  legal in G1
  }
  ```
- In Grammar 2, **`IsisEntity` and `IsisService` no longer support top-level `events`** - `IsisEvent` is only valid inside:
  - `IsisProperty` (`(features+=... | events+=...)`)
  - `IsisCollection` (`(features+=... | events+=...)`)
  - `IsisAction` (`(features+=... \| parameters+=... \| events+=...)`)

-> Therefore, `event UpdateNameDomainEvent` (top-level in entity) **must be removed** *unless* moved inside the action it relates to.

But the original code uses:
```java

action updateName(...)
```

In Apache Isis, `@Action(domainEvent=...)` binds an annotation to a domain event class - the event class name is *not* declared in the DSL; it's a Java type. So:
- `event UpdateNameDomainEvent` in DSL was **likely redundant** - just a declaration for IDE validation.
- Since Grammar 2 requires events to be scoped, and we cannot just delete them (constraint #1: avoid omitting elements), **we move it inside the `action`** - but *only if* it's semantically tied to that action.

 We'll move `event UpdateNameDomainEvent` *into* `action updateName { ... }`.
